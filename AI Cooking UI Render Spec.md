
# AI Cooking Assistant — Rendering Architecture Spec

**Version:** 0.2  
**Author:** Oleg Mrynskyi  
**Status:** Active  
**Last Updated:** April 2026

## Overview

This document specifies the rendering architecture for the agent-controlled canvas. The agent emits **typed, structured canvas operations** as JSONL. The React canvas renderer maps each operation to a React component placed in a named CSS grid zone. No HTML is generated by the agent. No iframe. No bleach sanitizer.

This is an A2UI-aligned design: a fixed component catalog constrains what the agent can render; operations stream progressively so the UI updates as the LLM generates, not after it finishes.

---

## Core Principle

```
Agent output  →  JSONL ops (type + data)
Render layer  →  React components mapped by type
Layout layer  →  9-zone CSS grid (default zone per type)
Streaming     →  Skeleton → Content events as LLM streams
```

The agent knows: component types, their data schemas, and the topological ordering rule.  
The agent does not know: CSS classes, HTML structure, zone names, or layout math.

## Op Format — JSONL

The agent outputs one JSON object per line. No outer array. Each line is independently parseable as it completes.

```
op      — add | update | remove | focus | move
id      — stable string identifier
type    — catalog type name (required for add)
data    — typed data object (required for add; partial for update)
parent  — parent component id (only for recipe-option)
zone    — new zone name (only for move)
```

### Examples

**Step view + progress bar + timer:**
```jsonl
{"op":"add","id":"step-1","type":"step-view","data":{"step_number":1,"total_steps":6,"recipe":"Pasta Carbonara","instruction":"Bring a large pot of salted water to a boil.","tip":"Use at least 4 litres — it should taste like the sea.","tags":["~10 min","stovetop"],"action":"next_step"}}
{"op":"add","id":"progress-1","type":"progress-bar","data":{"current":1,"total":6}}
{"op":"add","id":"timer-1","type":"timer","data":{"duration_seconds":600,"label":"Boiling","auto_start":true}}
```

**Advancing to step 2 (update existing IDs):**
```jsonl
{"op":"update","id":"step-1","data":{"step_number":2,"instruction":"Add pasta to the boiling water."}}
{"op":"update","id":"progress-1","data":{"current":2}}
```

**Recipe selection (parent before children):**
```jsonl
{"op":"add","id":"recipe-grid","type":"recipe-grid","data":{}}
{"op":"add","id":"r-1","type":"recipe-option","parent":"recipe-grid","data":{"title":"Pasta Carbonara","duration":"30 min","tags":["italian"],"action":"select_recipe_carbonara"}}
{"op":"add","id":"r-2","type":"recipe-option","parent":"recipe-grid","data":{"title":"Aglio e Olio","duration":"20 min","tags":["italian"],"action":"select_recipe_aglio"}}
{"op":"add","id":"r-3","type":"recipe-option","parent":"recipe-grid","data":{"title":"Cacio e Pepe","duration":"25 min","tags":["italian"],"action":"select_recipe_cacio"}}
```

**Suggestion + dismiss:**
```jsonl
{"op":"add","id":"suggestion-1","type":"suggestion","data":{"heading":"While you wait","body":"You could chop the garlic now — it'll save you time in step 3.","action_label":"Got it"}}
```
```jsonl
{"op":"remove","id":"suggestion-1"}
```

## Topological Ordering Rule

The agent MUST order ops so:

1. **Parents before children** — `recipe-grid` before all its `recipe-option` ops
2. **Most important first** — `step-view` before `timer`, `timer` before `suggestion`
3. **Within each data object — critical keys first** — `instruction` before `tags` before `action`

This ordering lets the streaming healer render skeletons and fill content in priority order.

**Reference ordering for common scenarios:**

| Scenario | Order |
|----------|-------|
| Step view | step-view → progress-bar → timer (if present) → suggestion (if present) |
| Recipe selection | recipe-grid → recipe-option × N |
| Warning | alert → (existing components untouched) |

## Canvas State Summary (Agent Input)

Before each render agent turn, the backend formats the current session canvas state as a compact text summary:

```
CURRENT CANVAS:
- step-1 (step-view): step 2/6 "Add pasta to the boiling water."
- progress-1 (progress-bar): 2/6
- timer-1 (timer): 600s Boiling [auto_start=true]
```

Empty canvas: `CURRENT CANVAS: empty`

The agent uses existing IDs to emit `update` ops. If an ID does not appear in the summary, the agent emits `add`.

## Streaming Architecture

### JSONStreamHealer

`render_agent/healer.py` — state machine over the LLM token stream.

**State:**
```python
depth: int          # brace nesting depth
in_string: bool     # currently inside a JSON string
escape_next: bool   # next char is escaped
line_buffer: str    # accumulates current object
skeleton_emitted: set[str]  # ids already skeleton-emitted
```

**Events:**
```python
@dataclass
class SkeletonEvent:
    id: str
    component_type: str

@dataclass
class ContentEvent:
    op: dict        # validated, ready to send
```

**Flow:**

1. Each token chunk is fed to `healer.feed(chunk)` → returns list of events
2. After `{` detected (`depth == 1`): scan buffer for `"type":"..."` and `"id":"..."` via regex
3. When both found → emit `SkeletonEvent` (once per id, idempotent guard)
4. As chars arrive: track `in_string`, `escape_next`; increment/decrement `depth` on `{`/`}` outside strings
5. When `depth` returns to 0 after a `{`: attempt `json.loads(line_buffer.strip())`
6. On success → emit `ContentEvent`; reset `line_buffer`
7. On failure → keep buffering (rare: malformed partial)
8. `healer.finalize()` → attempts parse of remaining buffer as fallback

### Graph streaming

`render_agent/graph.py` exposes:
```python
async def astream_events(
    intent: str,
    context: str,
    canvas_state: dict,
) -> AsyncGenerator[SkeletonEvent | ContentEvent, None]:
    ...
```

Uses `llm.astream()` internally. Feeds each token chunk to `JSONStreamHealer`. Yields events as they are emitted.

### WebSocket delivery

For each event, the backend sends an immediate WS message to the client:

```python
# SkeletonEvent
await ws.send_json({
    "type": "canvas_ops",
    "operations": [{"op": "skeleton", "id": ev.id, "type": ev.component_type}]
})

# ContentEvent
await ws.send_json({
    "type": "canvas_ops",
    "operations": [ev.op]
})
```

The existing frontend `canvas_ops` handler in `CanvasContext.tsx` processes these one at a time. No batching needed on the backend; the 16ms batching on the frontend handles rapid arrivals.

## React Canvas Renderer

### Zone grid

`Canvas.tsx` renders a CSS grid matching `design-system.css`:

```
┌────────────┬───────────────────────────┬────────────┐
│ corner-tl  │          top              │ corner-tr  │  76px
├────────────┼───────────────────────────┼────────────┤
│   left     │         center            │   right    │  flex
├────────────┼───────────────────────────┼────────────┤
│ corner-bl  │         bottom            │ corner-br  │  80px
└────────────┴───────────────────────────┴────────────┘
```

### Component file map

```
apps/web/src/canvas/
  Canvas.tsx              — zone grid; idle state (mic icon); dispatches to right zone
  components/
    Skeleton.tsx          — shimmer placeholder; knows zone by component_type
    StepView.tsx          — step-view (T-013)
    ProgressBar.tsx       — progress-bar
    Timer.tsx             — timer (T-014): countdown ring, tap pause/resume, pulse+audio at zero
    Suggestion.tsx        — suggestion (T-016): swipe-to-dismiss, action button
    Alert.tsx             — alert: standard + urgent variants
    RecipeGrid.tsx        — recipe-grid: 3-col grid, renders RecipeOption children
    RecipeOption.tsx      — recipe-option: tappable card
    IngredientList.tsx    — ingredient-list: scrollable rows
    Camera.tsx            — camera (T-015): permission, 3-frame capture, auto-remove
    TextCard.tsx          — text-card (T-017): markdown, 3-line truncate
```

### Skeleton lifecycle

When `skeleton` op arrives:
- Reducer stores `{ id, type, data: null, skeleton: true }` in canvas state
- `Canvas.tsx` renders `<Skeleton type={type} />` in the correct zone
- `Skeleton` shows a shimmer animation sized appropriately for the component type

When `add` op arrives for same id:
- Reducer upserts (replaces skeleton entry with real data)
- `Canvas.tsx` renders the actual typed component
- `animate-in` CSS class triggers fade + slide-up (180ms)

### 16ms render batching

`CanvasContext.tsx` buffers incoming WS ops with `requestAnimationFrame`:

```typescript
useEffect(() => {
  return subscribe((msg) => {
    if (msg.type === "canvas_ops") {
      pendingOps.push(...msg.operations);
      if (!rafPending) {
        rafPending = true;
        requestAnimationFrame(() => {
          rafPending = false;
          const batch = pendingOps.splice(0);
          batch.filter(validateOperation).forEach(dispatch);
        });
      }
    }
  });
}, [subscribe]);
```

### Upsert on duplicate add

```typescript
case "add": {
  const existing = next.get(op.id);
  const component: CanvasComponent = existing
    ? { ...existing, data: { ...existing.data, ...op.data } as CanvasComponent["data"] }
    : { id: op.id, type: op.type, data: op.data, focused: false };
  next.set(op.id, component);
  return next;
}
```

## System Prompt (Rendering Section)

The render agent system prompt includes:

```
COMPONENT CATALOG
Output one JSON object per line. Use only these types and data fields.

step-view: Renders the current recipe step in the center zone.
  Required: step_number (int), total_steps (int), recipe (str), instruction (str)
  Optional: tip (str), tags (str[]), action (str — action id for Next step button)
  Key order: step_number, total_steps, recipe, instruction, tip, tags, action

progress-bar: Shows step progress in the top zone.
  Required: current (int), total (int)

timer: Countdown timer in corner-br zone.
  Required: duration_seconds (int), label (str), auto_start (bool)

suggestion: Proactive tip in the bottom zone.
  Required: heading (str), body (str)
  Optional: action_label (str — button text)

alert: Warning strip in the top zone.
  Required: text (str)
  Optional: urgent (bool — default false)

recipe-grid: Recipe selection grid in center zone. No data fields.
  Children: emit recipe-option ops with parent="<this id>"

recipe-option: Single recipe card. Child of recipe-grid.
  Required: title (str), action (str)
  Optional: description (str), duration (str), tags (str[])

ingredient-list: Scrollable ingredient rows in center zone.
  Required: items (array of {name: str, qty: str})

camera: Camera capture in center zone.
  Required: prompt (str — analysis context)

text-card: Generic markdown card in center zone.
  Required: body (str — supports **bold** and _italic_)

CANVAS STATE
{canvas_state}

ORDERING RULE
Emit parents before children. Emit most important component first. Within each data object, emit critical keys first (instruction before tags before action).

OUTPUT FORMAT
One JSON object per line. No array brackets. No markdown. No explanation.
Use "update" for IDs that already appear in CANVAS STATE. Use "add" for new IDs.

EXAMPLE
{"op":"add","id":"step-1","type":"step-view","data":{"step_number":1,"total_steps":6,"recipe":"Pasta Carbonara","instruction":"Bring a large pot of salted water to a boil.","tags":["~10 min","stovetop"],"action":"next_step"}}
{"op":"add","id":"progress-1","type":"progress-bar","data":{"current":1,"total":6}}
```
